# LLM Agent 编排画布理解与展示方案

这次要做的不是一个“把几个方块连起来”的流程图，而是一个能说明我理解 AI Agent 如何工作的前端作品。它应该让访问者一眼看见：一个 Agent 不是单次问答窗口，而是一套会理解目标、拆分步骤、调用工具、读写记忆、检查结果并继续推进的执行系统。

## 我会先整理概念吗？

会，而且应该先整理。

原因很简单：Agent / 工作流可视化最容易做成空洞的视觉炫技。节点、连线、动效都可以很漂亮，但如果背后没有清晰的 LLM 工程概念，懂行的人会马上看出来它只是“流程图 UI”。我会先把 Agent、记忆、工具链这些概念拆成可以被界面表达的结构，再决定每个节点应该展示什么状态、每条边代表什么控制关系、失败和回溯应该怎么呈现。

我的判断是：这个 Demo 的核心不是证明“我会用 React Flow”，而是证明“我知道 LLM 应用为什么需要编排”。

## 我对 LLM Agent 的理解

LLM Agent 可以理解为一个带执行能力的语言模型系统。

普通 LLM 更像一个回答者：用户给问题，它基于上下文生成回答。Agent 则更像一个被赋予目标的执行者：它会先理解目标，再决定下一步要不要查资料、调用工具、读取历史、生成中间结果，必要时还会根据工具返回结果修正自己的计划。

我会把 Agent 拆成四层：

1. **目标层**：用户真正想完成什么。比如“帮我分析一个 PR 风险”，不是简单回答一句话，而是需要读取 diff、识别风险、整理建议。
2. **推理层**：LLM 根据当前上下文判断下一步动作。这里不是神秘的“智能”，更像是受 prompt、上下文和工具 schema 约束的决策过程。
3. **行动层**：Agent 调用外部能力，比如搜索、读取文件、请求 API、执行函数、写入数据库。
4. **反馈层**：工具结果回到 Agent，Agent 再判断是否继续、修正、终止或交给下一个 Agent。

所以在画布里，Agent 节点不能只显示一个头像或一个模型名。它应该显示它承担的职责、输入上下文、可调用工具、当前执行状态，以及它把什么结果交给下游。

## 我对长短时记忆的理解

Agent 的记忆不是“把所有聊天记录塞进 prompt”。

短期记忆更像工作台：它保存当前任务还需要用到的临时信息，比如用户这轮目标、上一步工具返回、当前计划、还没解决的错误。它应该是轻量的、会被不断刷新和压缩的，因为模型上下文窗口是有限的。

长期记忆更像资料库：它保存跨会话仍然有价值的信息，比如用户偏好、项目规范、历史决策、个人简历、常用技术栈、过往案例。长期记忆通常不会原样全部注入 prompt，而是通过检索、打分、摘要后，只把当前任务相关的部分放进上下文。

在这个作品里，我会把它设计成两类可视化节点：

- **Working Memory**：展示当前运行中的上下文摘要、任务计划、最近工具结果、token 占用。
- **Long-term Memory / RAG Store**：展示向量召回、个人知识库、项目资料、偏好档案，以及“被召回的片段如何影响当前 Agent 判断”。

这样能避免泛泛地写“支持记忆”，而是让用户看见记忆如何进入推理链路。

## 我对工具链的理解

工具链是 Agent 从“会说话”变成“能做事”的关键。

LLM 本身不能可靠地访问实时世界，也不能天然执行代码、读数据库、发请求或改文件。工具调用就是把这些能力用结构化接口暴露给模型：模型不直接“幻想一个结果”，而是选择一个工具、填入参数、等待真实返回，再基于返回继续生成。

我会重点展示三点：

1. **工具不是按钮，而是受约束的能力**：每个工具应该有明确的名称、入参、返回值和失败状态。
2. **工具结果会改变 Agent 的下一步**：例如搜索失败后走 fallback，代码审查发现高风险后转交 reviewer agent。
3. **工具调用需要边界**：包括权限、超时、重试、循环保护、结果校验，避免 Agent 无限调用或错误扩散。

在画布里，工具节点可以设计成带输入输出端口的“能力卡片”，比如 `web_search`、`read_file`、`run_test`、`vector_retrieve`、`create_summary`。当流程运行时，连线可以有光流动效，工具节点显示参数、耗时、状态和返回摘要。

## 我会如何展示这个 Demo

我会把它命名为 **Light-Agent Flow**：一个极简但有真实工程逻辑的 AI Agent 编排画布。

展示方式不是让用户从零搭复杂 workflow，而是先给一个高质量预设场景：**“让 AI Agent 帮我分析一个前端项目需求，并输出执行计划”**。这个场景贴合个人网站，也能自然展示前端与 LLM 结合能力。

推荐默认画布结构：

```text
User Goal
  -> Planner Agent
  -> Memory Router
      -> Short-term Context
      -> Long-term Knowledge / RAG
  -> Tool Executor
      -> Search / File Reader / Test Runner
  -> Critic Agent
  -> Final Response
```

每个节点的展示重点：

- **User Goal**：用户输入的真实目标，强调 Agent 从意图开始。
- **Planner Agent**：拆解任务步骤，展示计划不是一次性答案。
- **Memory Router**：决定读短期上下文还是长期知识库。
- **Tool Executor**：展示工具调用的结构化参数、返回值、失败状态。
- **Critic Agent**：检查输出是否满足目标，体现自检和回溯。
- **Final Response**：输出最终结果，并展示引用了哪些记忆和工具证据。

交互上我会做成“点击节点看侧边详情”的模式。画布负责建立整体理解，右侧 Inspector 面板负责解释节点内部发生了什么。这样既高颜值，也不会把复杂信息全部堆在画布上。

## 项目结构 Overview

当前项目是一个 React 18 + TypeScript + Vite 的个人网站，风格已经偏向“极简黑白 + 高级动效 + AI 产品感”。

现有结构可以这样理解：

```text
src/
  App.tsx
    负责全站骨架、Loader、导航、Hero、Projects、Dashboard 的组合

  components/
    HeroSection.tsx
      首页 AI 交互分身，模拟 RAG Enabled 和 Streaming UX

    ProjectSection.tsx
      作品集区域，当前 Light-Agent Flow 会出现在这里

  constants.ts
    动画曲线、文案数据、项目卡片与控制台日志

  context/
    LanguageContext.tsx
      中英双语切换状态

  index.css
    Tailwind 基础样式与少量滚动条工具类
```

如果要正式实现 Agent 编排画布，我建议新增一个独立模块，而不是继续把复杂画布塞进 `ProjectSection.tsx`：

```text
src/
  components/
    agent-flow/
      AgentFlowCanvas.tsx
      AgentNode.tsx
      ToolNode.tsx
      MemoryNode.tsx
      InspectorPanel.tsx
      flowData.ts
      types.ts
```

这样结构会更清晰：`ProjectSection` 只负责展示入口，真正的画布交给 `agent-flow` 模块维护。后续如果要换 React Flow、扩展节点类型、增加运行模拟，也不会污染首页其它区域。

## 展示风格建议

我建议延续当前项目的视觉语言：大面积中性色、强排版、细边框、圆角玻璃容器、低饱和高亮色。不要做成 Dify / LangFlow 的后台工具风，而是做成“个人作品集里的 AI 操作台”。

具体风格可以这样落地：

- **整体底色**：深色画布区，和 `ProjectSection` 的 `bg-neutral-950` 保持一致。
- **节点设计**：白色或深灰玻璃卡片，顶部用小号 mono 标签标识 `AGENT`、`MEMORY`、`TOOL`。
- **连线设计**：默认细灰线，运行时出现 emerald / cyan 光流，表示上下文正在传递。
- **状态表达**：`idle`、`running`、`success`、`blocked`、`retrying`，每种状态有明确颜色和微动效。
- **信息层级**：画布只放关键摘要，详情放在右侧 Inspector，避免 UI 像概念海报一样拥挤。
- **动效气质**：使用 Framer Motion 的弹簧过渡，让节点激活、详情面板切换、日志滚动都有轻微呼吸感。

## 我会避免的“AI 味”

我不会在界面上堆很多大词，比如“自主智能”“无限进化”“全栈认知引擎”。这些词看起来热闹，但不说明工程能力。

我更想展示具体判断：

- Agent 为什么需要规划，而不是直接回答。
- 记忆为什么要分短期和长期，而不是无限塞上下文。
- 工具为什么要有 schema、权限和失败处理。
- 工作流为什么需要可观测状态，而不是黑盒等待。
- 前端为什么重要：因为用户需要看见 AI 正在做什么、卡在哪里、依据是什么。

这个 Demo 最终要传达的个人理解是：好的 AI 前端不是给模型套一个漂亮壳子，而是把模型、上下文、工具和状态变成用户能理解、能控制、能信任的产品体验。
